# TSPlay Workbench V1 落地任务清单

> 目标：把“已授权 Web 系统认知 + 接口发现 + 数据编排”的产品定义，拆成可以直接开工的功能清单和迭代顺序。

## 1. 目标闭环

V1 要跑通的最小闭环是：

`登录站点 -> 复用 session -> 探索主要页面 -> 捕获主要接口 -> 生成页面/API 卡片 -> 用户输入数据需求 -> AI 生成 Flow -> 本地执行 -> 输出基础分析`

## 2. 现有基础能力与缺口

### 已有基础能力

当前仓库已经具备这些可复用底座：

- session 保存、列出、导出、删除
- `observe_page`
- `draft_flow / finalize_flow / validate_flow / run_flow / repair_flow_context`
- `http_request / json_extract`
- artifacts、截图、HTML、DOM snapshot
- CLI / MCP Server 两套入口

### 关键缺口

V1 真正需要补的，不是新的浏览器引擎，而是这几类能力：

- 站点级探索，而不是单页观察
- 面向产品的只读探索模式
- 可持续记录的请求采集与页面动作关联
- 页面卡片 / API 卡片 / 实体卡片的生成逻辑
- 本地知识库存储和检索
- 基于知识库的 AI 编排
- 本地工作台 UI

## 3. 优先级划分

### P0

必须完成，否则闭环跑不通。

### P1

建议在 P0 跑通后尽快补齐，否则演示价值不强、用户体验较弱。

### P2

增强项，适合第二轮或验证期后推进。

## 4. P0 功能清单

### 4.1 `tsplay core` 与 Runtime

- [ ] `session` 接入到工作台
  验收：Workbench 能列出、选择、测试、删除已保存 session。
- [ ] 站点配置模型
  验收：支持配置 `site_id / start_url / allowed_domains / session_name`。
- [ ] 只读探索运行模式
  验收：默认不提交表单、不点击高风险按钮、不越过域名白名单。
- [ ] 站点探索入口
  验收：可从起始页开始抓取主菜单、链接、标题、表单、按钮、表格摘要。
- [ ] 探索结果 artifact 落盘
  验收：每次探索都能保存截图、DOM 摘要、页面 JSON 结果。
- [ ] 请求采集器基础版
  验收：能记录页面探索和用户动作触发的 `XHR / Fetch` 请求元数据。
- [ ] 页面动作与请求关联
  验收：至少能把“页面 URL + 触发动作 + 请求列表”串起来保存。
- [ ] 请求结果摘要化
  验收：能保存 method、URL、状态码、headers 摘要、body schema 摘要，不保存敏感原文。

### 4.2 Discovery Layer

- [ ] `Site Explorer`
  验收：从首页或菜单页识别主要导航结构，输出页面列表。
- [ ] `DOM Summarizer`
  验收：输出页面标题、菜单路径、表单字段、按钮名、表格列摘要。
- [ ] `Network Recorder`
  验收：探索期间自动记录 XHR / Fetch 请求。
- [ ] `Schema Inferencer`
  验收：对请求和响应样本推断基础 schema。
- [ ] `Risk Classifier`
  验收：接口和动作至少区分 `read / read_download / write / critical`。

### 4.3 Knowledge Layer

- [ ] 本地知识库存储
  验收：至少能保存 `sites / pages / apis / entities / runs`。
- [ ] 页面卡片生成器
  验收：为每个页面生成标题、菜单路径、摘要、动作列表、风险等级。
- [ ] API 卡片生成器
  验收：为每个接口生成 method、path、触发来源、schema、风险等级。
- [ ] 实体卡片基础版
  验收：能从主要响应结构中提取实体名和字段清单。
- [ ] 本地全文检索
  验收：能按页面标题、菜单名、接口路径、字段名检索卡片。

### 4.4 AI 层

- [ ] 意图解析器基础版
  验收：能从一句自然语言里提取目标、时间范围、对象、过滤条件。
- [ ] 知识检索器
  验收：能根据用户意图返回候选页面卡片和 API 卡片。
- [ ] 执行策略选择器
  验收：能在 `ui_first / api_first / hybrid` 之间做基础判断。
- [ ] Flow 生成器基础版
  验收：能生成一条可执行的 `TSPlay Flow`。
- [ ] 结果摘要器
  验收：能把执行结果和基础统计总结成人话。

### 4.5 VSCode / Workbench

- [ ] 站点接入页
  验收：能新增站点配置、选择 session、启动探索。
- [ ] 页面地图页
  验收：能查看页面列表、页面卡片和最近探索结果。
- [ ] 接口列表页
  验收：能查看接口卡片、schema 和触发来源。
- [ ] 数据任务页
  验收：能输入自然语言需求、查看 AI 计划、触发执行。
- [ ] 结果与日志页
  验收：能查看执行日志、数据结果表和分析摘要。

## 7. 推荐实施顺序

### 里程碑 M1：登录态与探索跑通

目标：

- 本地启动 Workbench
- 选择已保存 session
- 启动只读探索
- 输出页面列表和原始 artifact

对应任务：

- `session` 接入
- 站点配置模型
- 只读探索模式
- `Site Explorer`
- 基础页面结果落盘

### 里程碑 M2：接口发现与知识卡片跑通

目标：

- 探索期间采集请求
- 生成页面卡片和 API 卡片
- 本地可检索

对应任务：

- `Network Recorder`
- `Schema Inferencer`
- 页面卡片生成器
- API 卡片生成器
- 本地知识库存储

### 里程碑 M3：AI 编排与 Flow 执行跑通

目标：

- 用户输入自然语言任务
- AI 基于知识卡片生成 Flow
- 执行并输出结果

对应任务：

- 意图解析器
- 知识检索器
- 执行策略选择器
- Flow 生成器
- 数据任务页
- 结果与日志页

### 里程碑 M4：基础分析与演示增强

目标：

- 数据结果结构化
- 输出基础分析和人话总结
- 补全 artifact 查看和 Flow 预览

对应任务：

- 结果摘要器
- 数据结果表
- Flow 双视图
- artifact 预览

## 8. 建议分工

### 方向 A：`tsplay core`

负责：

- 只读探索模式
- 请求采集
- 页面结果和请求结果 artifact
- 对外 CLI / MCP / 本地 API

### 方向 B：知识与编排

负责：

- 卡片模型
- 本地知识库存储
- 检索
- 意图解析
- Flow 生成

### 方向 C：Workbench

负责：

- 站点接入页
- 页面地图页
- 接口列表页
- 数据任务页
- 结果与日志页

## 9. 第一批建议直接开工的任务

如果明天就开始动手，我建议先开这 8 项：

1. [ ] 站点配置模型与本地存储
2. [ ] Workbench 接入现有 `list-sessions / save-session / get-session`
3. [ ] 只读探索模式最小实现
4. [ ] 单次站点探索结果 JSON artifact
5. [ ] XHR / Fetch 请求采集最小实现
6. [ ] 页面卡片与 API 卡片的数据结构
7. [ ] 本地知识库最小表结构
8. [ ] 数据任务页最小版：输入需求 -> 返回候选页面/API -> 生成 Flow

## 10. Definition of Done

只有同时满足下面这些条件，V1 才算真正跑通：

- 能成功登录并复用 session
- 能识别 80% 以上主要菜单页面
- 能捕获主要 XHR / Fetch 接口
- 能生成可读页面卡片和 API 卡片
- 能根据自然语言生成一条可执行 Flow
- 能完成至少一次数据获取、结构化落地和基础分析输出
