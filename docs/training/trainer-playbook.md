# TSPlay 讲师手册

这份手册服务于内部讲师、Enablement 负责人和项目带教。目标很简单：让培训可复制，而不是每次都靠临场发挥。

## 开课前

### 提前 7 天

- 确认学员名单和角色分布
- 决定本次课程是“半天入门”还是“2 天 Bootcamp”
- 选定 Labs 和 Capstone 场景
- 确认本地或内网环境能访问 `demo/` 页面

### 提前 1 天

- 演练所有例子和 Labs
- 确认示例脚本、Flow、artifact 路径无误
- 准备评分表和实验报告模板
- 让学员完成 Day 0 预习

### 开课当天

- 先做环境检查，不要先讲大理论
- 提前指定助教负责排查环境问题
- 保留 20% 时间给实操和答疑

## 授课原则

- 每个新概念都要对应一个可运行例子
- 每个小时至少安排一次操作
- 先讲“什么时候用”，再讲“怎么写”
- 先让学员跑通，再讲优化
- 评审时优先说结构、稳定性和恢复策略，不只说“能不能跑”

## 推荐授课顺序

1. 用根目录 README 建立全局认知
2. 用 CLI 和 Lua 建立“动作感”
3. 用 Flow 建立“结构化思维”
4. 用 MCP 建立“AI 与自动化协作思维”
5. 用 Capstone 验证是否能真正交付

## 常见卡点与处理方式

| 卡点 | 常见表现 | 讲师处理建议 |
| --- | --- | --- |
| 不会找 selector | 一直复制浏览器临时 selector | 让学员先看 `observe_page` 的候选 selector，再解释稳定性 |
| 滥用 `sleep` | 所有等待都写成固定暂停 | 强制改成 `wait_for_selector`、`retry` 或 `wait_until` |
| Flow 过长过散 | 逻辑堆在一起、变量命名混乱 | 要求补 `name`、`vars` 和设计说明 |
| 只会跑，不会查 | 报错后只会重试 | 带着看 trace、HTML、DOM snapshot |
| 不理解授权 | MCP 下一上来就要全开 | 让学员按动作解释每个授权的风险 |

## 评审建议

- 先看交付物是否齐全
- 再看结构是否清楚
- 然后看 selector、控制流和恢复设计
- 最后再看风格和命名

讲师最好避免现场直接替学员写答案，更好的方式是：

- 指出哪一步的假设不成立
- 指出哪份 artifact 最值得看
- 让学员自己解释为什么用了某个动作

## 训练营后的动作

- 收集所有学员提交物
- 记录每个 Lab 的通过率
- 把常见问题沉淀为 FAQ 或新示例
- 识别可以培养为讲师的人
- 更新下一轮课程用到的示例、评分表和节奏

## 版本维护建议

当 TSPlay 增加新动作或 MCP 工具时，讲师应同步检查：

- 根目录 README 是否需要更新
- Labs 是否应该新增或替换
- 评分维度是否要调整
- Capstone 是否要增加新场景

培训材料和产品代码要一起演进，不能把课程当成一次性文档。
