# TSPlay 考核与认证

培训不是讲完就算完成。这里给出一套简单但可执行的考核体系，用来判断学员是否已经具备实际交付能力。

## 考核结构

| 类型 | 用途 | 建议时点 |
| --- | --- | --- |
| 诊断考核 | 判断学员处在 L0-L1 还是已具备基础 | 开课前 |
| 实验考核 | 看学员是否掌握当日核心动作 | 每个 Lab 后 |
| Capstone 结业考核 | 看学员是否能整合能力独立交付 | 训练营结束 |
| 复盘考核 | 看学员是否能把能力迁移到真实业务 | 训练营后 2-4 周 |

## 评分维度

每个维度按 0-3 分评分：

| 维度 | 0 分 | 1 分 | 2 分 | 3 分 |
| --- | --- | --- | --- | --- |
| 环境与执行 | 无法运行 | 需要大量协助 | 基本能独立运行 | 能独立排查常见问题 |
| selector 质量 | 大量脆弱定位 | 可运行但不稳 | 大多数定位合理 | 定位稳定且可解释 |
| Flow 结构 | 混乱、难审查 | 勉强可读 | 结构清楚 | 结构稳定且可复用 |
| 健壮性设计 | 完全没有 | 只会加 `sleep` | 能用 `retry/if` 等 | 能结合 `on_error/wait_until` 做局部恢复 |
| 数据与变量 | 无变量设计 | 变量零散 | 会用 `save_as` / `set_var` | 变量设计清晰且支持复用 |
| 安全边界 | 不理解授权 | 知道但常误用 | 能按动作申请授权 | 能解释风险和边界 |
| 观察与修复 | 不会看 trace | 只会看报错 | 能定位 artifact | 能利用 repair 上下文系统修复 |
| 表达与交付 | 无法解释方案 | 只能描述步骤 | 能解释设计原因 | 能做可复盘、可讲授的交付说明 |

## 晋级建议

### L1 通过

- 总分至少 10
- “环境与执行”不能为 0
- 完成 Labs 1-2

### L2 通过

- 总分至少 14
- “Flow 结构”与“数据与变量”至少 2 分
- 完成 Labs 3-4

### L3 通过

- 总分至少 18
- “健壮性设计”至少 2 分
- 完成 Lab 5 和 1 个 Capstone

### L4 通过

- 总分至少 20
- “观察与修复”“安全边界”至少 2 分
- 完成 Lab 6 或 Stretch Lab

### L5 通过

- 总分至少 22
- 主持或协助 1 次课程
- 至少评审 3 份学员作品

## 证据清单

每次评审建议至少收集这些证据：

- Flow 或 Lua 脚本
- `validate_flow` 或 `run_flow` 输出
- `artifacts/` 里的截图或 HTML
- 实验报告
- 讲师评语

## 认证建议

| 认证名称 | 对应等级 | 建议发放条件 |
| --- | --- | --- |
| TSPlay Operator | L1-L2 | 能独立运行并维护基础 Flow |
| TSPlay Flow Author | L3 | 能设计健壮 Flow 并处理异常 |
| TSPlay MCP Integrator | L4 | 能用 MCP 工具链完成草拟、执行和修复 |
| TSPlay Coach | L5 | 能授课、评审、维护训练素材 |

## 复盘机制

训练营结束后 2-4 周建议再做一次复盘：

- 学员是否把能力迁移到了真实业务页面
- 团队是否沉淀出新的通用 Flow 模板
- 哪些 Lab 过难或过浅
- 哪些页面需要新增到 `demo/` 或 `script/`

讲师评审模板见 [templates/capstone-review.md](templates/capstone-review.md)。
