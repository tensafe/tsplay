# TSPlay Flow 中文业务场景模板

## 这份文件适合什么时候看

当用户知道业务目标，但还没组织好一句“能让 Codex 直接开始写 Flow”的中文提问时，先看这份文件。

这些模板的目标不是替用户写死细节，而是把需求整理成最容易直接落成 Flow 的输入形状。

## 通用标准问法

```text
帮我写一条 TSPlay Flow。
- 页面: <URL 或本地页面>
- 目标: <要完成的业务动作>
- 输入: <关键词 / 文件 / 条件 / 行号，没有就写无>
- 输出: <save_as / JSON / CSV / Excel / artifact 路径>
- 授权: <readonly / browser_write / full_automation / allow_*>
- 要求: 结果要可 review，变量名和 artifact 路径写清楚
```

## 模板 1: 登录

```text
帮我写一条 TSPlay Flow 处理登录。
- 页面: <登录页 URL>
- 目标: 登录成功后进入 <目标页面或目标状态>
- 输入: 账号、密码、验证码要求、角色
- 输出: 登录成功的断言结果，必要时保存会话或 storage state
- 授权: <最小权限>
- 要求: 如果后续流程会反复用到登录态，优先考虑顶层 `browser.use_session` 设计
```

适用场景：

- 首次写登录 Flow
- 要把一次性登录改成会话复用

## 模板 2: 搜索

```text
帮我写一条 TSPlay Flow 做搜索。
- 页面: <搜索页 URL>
- 目标: 输入关键词后完成搜索，并验证结果区域出现
- 输入: 搜索关键词
- 输出: 结果区域状态、结果数、标题或链接
- 授权: <最小权限>
- 要求: 先保证搜索能跑通，再提取关键结果
```

适用场景：

- 搜索框输入
- 条件筛选
- 搜索后抓摘要或链接

## 模板 3: 导入

```text
帮我写一条 TSPlay Flow 做批量导入。
- 页面: <导入页 URL>
- 目标: 把 CSV 或 Excel 数据逐行导入页面
- 输入: <文件路径>，必要时补 sheet、range、起始行
- 输出: 成功失败结果台账，写到 JSON 或 CSV
- 授权: 需要文件读写时给出对应权限
- 要求: 用 `foreach` 处理每一行，必要时用 `on_error` 保证单行失败不拖垮整批
```

适用场景：

- CSV 导入
- Excel 导入
- 断点续跑导入

## 模板 4: 导出

```text
帮我写一条 TSPlay Flow 做导出。
- 页面: <导出页 URL>
- 目标: 点击导出或下载按钮，确认导出成功
- 输入: 查询条件、筛选条件、时间范围
- 输出: 导出成功状态，必要时保存文件路径或后续校验结果
- 授权: 如果涉及下载文件，说明文件权限
- 要求: 先断言导出按钮和结果状态，再处理文件落地或后续校验
```

适用场景：

- 报表导出
- 文件下载
- 导出后再对文件做二次验证

## 模板 5: 表格抓取

```text
帮我写一条 TSPlay Flow 抓取表格。
- 页面: <URL>
- 目标: 抓取表头和所有行
- 输入: 表格 selector 如果已知就给出，不知道就写未知
- 输出: 保存到变量、JSON、CSV，或三者都要
- 授权: <最小权限>
- 要求: 如果表格结构稳定，优先考虑 `capture_table`；如果 selector 不确定，优先说明是否要先 observe_page
```

适用场景：

- 列表页抓数
- 抓表格后再做比对
- 抓表格后写入别的系统

## 模板 6: MCP Finalize

```text
帮我按 TSPlay MCP 的 finalize 思路把需求收敛成 Flow。
- 页面: <URL>
- 意图: <自然语言业务需求>
- 输入: <变量、关键词、文件、条件>
- 输出: <期望的 Flow 结果或 artifact>
- 授权: <readonly / browser_write / full_automation>
- 要求: 优先走 `finalize_flow` 的思路；如果遇到 needs_input、needs_permission、needs_repair，再明确下一步怎么补
```

适用场景：

- 用户不会手写 Flow
- selector 不确定
- 需要先用 MCP 收敛，再修 Flow

## 模板 7: 邮件通知

```text
帮我写一条 TSPlay Flow 发送邮件通知。
- 目标: <成功通知 / 失败告警 / 跑完后发汇总 / 发附件报告>
- 收件人: <邮箱字符串或邮箱列表>
- 主题: <邮件主题>
- 内容: <正文文本或 HTML>
- 附件: <无，或要发送的文件路径 / 生成后的 artifact>
- SMTP: <用 connection 读取环境变量，或直接给 inline smtp 配置>
- 授权: <allow_email；如果有附件，再补文件权限>
- 要求: 结果要保存到 `email_result` 之类的清楚变量名
```

适用场景：

- 导入完成后通知
- 失败后发告警
- 生成 Excel 或 CSV 后发附件

## 模板 8: 修已有 Flow

```text
帮我修这条 TSPlay Flow。
- 文件: <flow 文件路径>
- 页面: <相关页面 URL，可选>
- 问题: <报错、超时、断言失败、变量不对、权限不够>
- 预期: <修完后应该得到什么结果>
- 限制: <不要改业务意图 / 不要转 Lua / 保持 artifact 路径>
```

适用场景：

- selector 失效
- 登录态过期
- 导入导出结果不对
- MCP finalize 后需要落到具体修复

## 模板 9: 做成教程风格

```text
帮我按 TSPlay 教程风格写一条 Flow。
- lesson 编号: <例如 161>
- 页面: <URL 或 demo 页面>
- 主题: <登录 / 搜索 / 导入 / 导出 / 表格 / 会话 / MCP>
- 目标: <要讲清楚的核心动作>
- 输出: <artifacts/tutorials/... 路径>
- 要求: `name`、`description`、`save_as`、artifact 路径都按教程风格写清楚
```

适用场景：

- 新增教程
- 把业务流程沉淀成教学素材

## 模板 10: 先排查再修

```text
帮我先排查这条 TSPlay Flow，再决定怎么修。
- 文件: <flow 文件路径>
- 页面: <URL，可选>
- 现象: <超时 / 找不到 selector / assert_text 失败 / 输出为空 / needs_repair>
- 目标: 找出最小修复方案
- 要求: 优先最小修复，不要直接整条重写
```

适用场景：

- 已有 Flow 很接近可用
- 团队更关心最小修补成本

## 一句话使用建议

如果你不知道该怎么问，先用“页面 + 目标 + 输入 + 输出 + 授权”这 5 个字段；如果你已经有 Flow 出错，就换成“文件 + 问题 + 预期 + 限制”这组字段。
